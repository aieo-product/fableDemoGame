# Phase 3 作業ログ: レビュー/検証（ワークフロー `katamari-review`）

**手法**: 4次元並列レビュー（正確性/パフォーマンス/ゲームプレイ/モジュール間契約）→ 指摘1件ごとに独立した懐疑エージェントが「反証を試みる」敵対的検証 → 確定指摘+既知課題を修正エージェントが一括適用 → 最終バリデータがビルド+ブラウザ実機プレイテスト。

**実行実績**: エージェント36体 / サブエージェント消費 約2,552,748トークン / 所要 約54分

**結果**: 確定指摘 21件 / 棄却 9件 / 修正 16件 / スキップ 4件 / 最終検証 合格

## 敵対的検証で棄却された指摘（偽陽性の排除）

検証エージェントは各指摘のコードを実際に読み、到達可能性・数値根拠・仕様適合を検査して反証した。代表例:

### ✗ ?r= dev start: boot preloads the world BEFORE applying the start radius — frame-1 mass-absorb of stale tier-0 content（correctness）

The surface facts check out: boot (main.js:245-248) preloads before ballPhys.reset while resetWorld (main.js:203-213) does the reverse, ?r=120 maps to radiusSim=1200, and the first PLAYING substep does run absorb.resolve with ballR=1200 against dense tier-0 content (maybeTierUp rescales only once per frame, after the substeps), producing a real frame-1 mass-absorb with combo/count/sluggish pollution. However, the finding's causal mechanism and suggested fix do not hold up. Spawn content depends only on tierIndex and the spawner's _scaleExp (both 0 in either order), not on ballRadiusSim — so applying the suggested fix (reset first, then preload) yields the same tier-0 world around the origin and the identical mass-absorb on the first substep. It is actually marginally worse: with ballRadiusSim=1200 the prewarm threshold (progress >= PREWARM_FRACTION at spawner.js:359-364) triggers, loading the N+2 band whose objects (r <= ~75 sim, all under 0.65*1200) are absorbed too; the sub-pixel sweep/despawn drain are amortized (64/frame) and never run during title frames, so they cannot mitigate. The 'correct order' path is already exercised — restart after win with ?r= still set runs resetWorld and exhibits the same mass-absorb — proving the boot/reset divergence causes no behavioral difference for the claimed symptom. The mass-absorb is inherent to the ?r= catch-up design (one rescale per PLAYING frame applied after absorb), which the finding itself defers to a separate finding. What remains is a consequence-free consistency nit (boot should call resetWorld() instead of duplicating a divergent sequence) on a dev-only URL key unreachable in normal gameplay (smoke test at ?r=120 passed clean). Medium severity is inflated; the proposed fix would not change the behavior complained about.

### ✗ Spawn-queue overflow loses placements until the next chunk-boundary crossing; _drainSpawn retry-push failure drops them permanently（correctness）

The code paths cited are real: _enqueueChunkSpawns (spawner.js:728-736) does return silently when _spawnQ.push fails, and top-up only occurs via the enqueuedThrough<wantK check in _recomputeBand (line 656), which runs only on chunk-boundary crossings or cache invalidation. The finding already concedes its case (2) (_drainSpawn retry-push, lines 805-812) is unreachable — after a shift there is always a free slot. Case (1) fails on reachability arithmetic: the ring is IntRing(8192) (spawner.js:239), while the worst-case single-frame flood — a tier change recomputing all three bands — enqueues only ~3,064 entries (band 0: 29 chunks within loadRadius 96 / cell 32 × objectsPerChunk 72 = 2,088; scenery bands: ~61 chunks × 8 = ~488 each), i.e. ~37% of capacity. The claim that 'a tier change floods the 8192-ring' is false by ~2.7x. Live entries are bounded by the wanted set (~3,064) because enqueuedThrough is monotone per record generation; filling the remaining ~5,100 slots would need thousands of stale entries trapped behind a blocked head, but _drainSpawn consumes stale entries without spending budget even under ALIVE_TOTAL_BUDGET back-pressure, the head record goes stale as the ball moves, and the chunk churn required to create stale entries simultaneously drives the 64/frame despawn drain and 32/frame sub-pixel sweep that release alive budget and resume draining within ~1s. Even in the hypothetical overflow, new chunks are enqueued nearest-ahead-first (insertion sort, lines 668-686), so dropped placements would be the farthest rim chunks, which tiers.js line 151 guarantees lie beyond the fog wall (FOG_FAR_K < loadRadiusSim - cellSizeSim) — invisible while idling, and topped up by line 656 on the first 32-sim-unit chunk crossing toward them. No visible population shortfall is reachable in actual gameplay; this is at most a defensive-coding suggestion.

### ✗ Win-screen idle camera orbits the world origin, not the ball — shows empty fog after long runs（correctness）

The code-level claims are accurate: updateIdle (src/render/cameraRig.js:246-258) orbits the world origin and looks at (0, r*0.5, 0); main.js:271-279 calls it during WIN; the ball can be up to REBASE_DISTANCE_SIM=1500 sim units from origin at win time; fog far = 55*r (~27.5-137.5 sim). However, the claimed player-visible failure is unreachable: #win-overlay (index.html:126-136, .overlay class) is position:fixed inset:0 z-index:20 with a fully opaque background (linear-gradient of opaque #0b0c14/#131630/#1c1438 under the radial layer), shown synchronously in the same frame as GAME_WIN (screens.js:101), with no fade or translucency (.hidden is display:none !important). The title overlay is the same opaque class, so the idle camera's framing is never visible in either non-playing state. The "empty fog" celebratory view cannot be observed in actual gameplay; the mis-centered orbit only renders dead pixels behind an opaque DOM layer. It is at most a latent issue that would matter only if the overlay were later made translucent.

### ✗ Ball and Effects cosmetic RNGs are not re-seeded on game:reset — same-seed runs are not reproducible（correctness）

The mechanical facts are accurate: Ball.reset() (src/render/ball.js:528-554) does not re-seed this._rng (created once at ball.js:163), and Effects.reset() (src/render/effects.js:385-392) does not re-seed this._rng (effects.js:252), while absorb.js does re-seed (src/physics/absorb.js:298). However, the claimed failure — 'same-seed runs are not reproducible', violating the determinism contract — does not hold. (1) The contract documented in spawner.js's header covers world generation only: chunk contents are a pure function of hash(worldSeed, cx, cz, band); the win-screen 'shareable ?seed=' (screens.js:32-34) shares the world layout, which is fully honored independent of these cosmetic RNGs. (2) The Ball RNG is consumed per absorption event, whose order depends on live player input — never replayable — so two same-seed runs diverge in these draws regardless of re-seeding. (3) The Effects RNG is consumed in Effects.update(dt, ...) where dt is explicitly the render-frame delta (effects.js:284; called with frameDt in main.js:276/315), making particle outcomes dependent on rAF timing and frame drops — non-deterministic even on a fresh load; the suggested fix cannot deliver reproducibility. (4) The absorb.js parallel is a category error: its RNG selects which objects are knocked off and re-enter the world (gameplay/sim state), hence must reset; Ball/Effects draws affect cosmetics only, matching the project invariant. (5) The screenshot-diff regression harness is hypothetical (none exists in the repo) and would be defeated by frame-timing-dependent effects emission anyway. What remains is a harmless cross-restart state carryover in observably-random cosmetic streams with no contract violation and no player-detectable defect. A one-line re-seed in each reset() would be acceptable hygiene, but it fixes no real behavior.

### ✗ Effects QuadPool rewrites and uploads the full instanceMatrix buffer every frame any particle is alive, with per-particle Matrix4/Quaternion compose（performance）

The mechanical observation is partially real but the finding's central quantitative claim is wrong, and the impact claim does not survive scrutiny. (1) FALSE headline: "all three pools upload their whole buffers (≈12KB/frame) whenever even one speed-line is alive" — each QuadPool.update early-returns when its own aliveCount===0 (effects.js:155) and needsUpdate is per-pool, so the common rolling case touches only the 32-slot lines pool: 2KB/frame, not 12KB. The 12KB case needs pops+ring+lines simultaneously alive, a sub-second window around tier-ups. (2) Intentional and documented: the method's doc comment (effects.js:149-150) explicitly states "small caps — full needsUpdate per touched frame is cheap" — a deliberate tradeoff, not an oversight. (3) The binding DESIGN.md mandates updateRanges/partial uploads only for the WORLD InstancedPools (capacities 128-512, 24+ meshes, "P2 risk mitigation"); effects are specced merely as "~3 effect meshes, instanced, pooled" — instances.js complies where required. (4) "Measurable on mobile" is unsupported: 2KB (worst 12KB) bufferSubData plus typically ≤32 (worst ≤192) generic Matrix4.compose calls is sub-0.05ms; the in-browser smoke test passed cleanly and the spec's dynamic-resolution governor absorbs far larger costs. (5) No project invariant is violated — zero per-frame allocation holds via module-level scratch objects. What survives is an optional micro-optimization note with no measurable payoff, not a performance defect.

### ✗ T0 pickups miss inside the ball's visual silhouette — add planar pickup-forgiveness contact for absorbables（gameplay）

The finding's geometry is arithmetically correct (verified: coin simR=0.12, restY=0.012, ball center y=R=0.5 gives planar pickup half-width 0.382 vs 0.62 top-down overlap), but it fails as a defect. (1) The code implements the binding DESIGN.md spec exactly — line 16 mandates static bounding spheres with a single overlap test dispatching absorb-vs-pushback; the proposed cylinder-XZ contact is a spec change, not a bug fix. (2) The 'visual silhouette' baseline (R+r planar) is top-down shadow overlap; the ball's silhouette edge there is its equator 0.5 sim above ground, and the rendered ball surface cannot touch a ground coin at that distance. Honest rendered contact for the coin is ~0.27 planar, so the current 0.382 threshold is already ~40% MORE generous than what the player sees touching from the gameplay camera — 'visibly grazes and gets nothing' only holds in a top-down view the game doesn't use. (3) The 0.281 absorbs/s Monte-carlo is the finder's own non-reproducible simulation; absorb pacing is a tuning outcome and the smoke test passed the full loop. (4) Nothing is broken, unreachable, or progression-blocking; the test is continuous in radius (seamlessness law intact). Pickup forgiveness is a legitimate optional game-feel polish suggestion, severity low at most — 'critical' is heavily inflated.

### ✗ Knock-off ejects objects without removing their volume — growth double-counts on re-absorb（gameplay）

The described mechanism is real (knockOff at ball.js:400-428 never decrements radiusSim; spawner.reinject re-creates ALIVE re-absorbable instances; radiusSim is written only by absorb growth, rescale, and reset), but the finding fails as a defect. (1) The behavior is exactly what the binding DESIGN.md specifies: knock-off is defined as ejecting 1-3 newest stuck objects 're-entering the world as re-absorbable instances' as a feel-layer beat, with growth defined only on absorb and no refund or HUD decrement anywhere in the spec — the code conforms to its authority. (2) The 'farm-by-bonking exploit' confers no advantage: re-absorbing an ejected object grants the same growth as absorbing any equal-size fresh object, the world is kept saturated with fresh spawns, and the loop requires ramming a too-big object above 0.7*speedCap (losing momentum to the restitution bounce) then chasing randomly-popped objects — strictly slower than normal play. (3) The quantitative support is fabricated: the finding cites GROWTH_K=10; the actual value is GROWTH_K=0.45 (tuning.js:66, matching DESIGN.md), and the +4% figure applies only to maximum-ratio (0.65R) objects, not a 'recycled coin'. (4) The finding itself concedes the pushback tuning works because the loss is cosmetic and offers 'intended kindness' as a valid alternative — which DESIGN.md confirms is the intent. Residual conservation-of-volume incoherence is a design-philosophy nit, not a code bug.

### ✗ Hysteresis down-shift emits a celebratory 'tierUp' and orphans spawner records of band tier+3（contracts）

The static code reading is accurate on both points: scaleManager.js:168-172 does emit the same EVT.TIER_UP on the downward hysteresis branch (and all five cosmetic consumers — hud, environment, cameraRig, sfx, effects — treat it as an upgrade celebration with no direction check), and spawner.js's machinery (_onTierChange unloads only band <= newTier-2, update's loop covers bands tier..tier+2, _cleanupLeftovers skips rec.band >= tier) would indeed strand band-T+2 pre-warm records after a T -> T-1 shift. However, the failure scenario is unreachable in actual gameplay: ball.radiusSim is strictly non-decreasing in play (absorb.js:201 cbrt growth is the only in-run mutation; knock-off in absorb.js/render/ball.js removes cosmetic stuck objects without touching radiusSim; reset() pairs with scaleManager.reset() which sets tierIndex=0 directly). The rescale preserves trueRadius to ~1 ulp (radiusSim *= 0.2, worldScale /= 0.2), while the down-shift requires trueRadius to fall more than TIER_HYSTERESIS = 10% (tuning.js:25) below a threshold it already crossed — a 1e-16-relative float perturbation cannot cross a 10% band, so even the finding's own 'float-edge' trigger does not exist. Debug paths (forceRescale, ?r= start radius) also cannot lower trueRadius. The down branch is dead defensive code, correctly self-documented as 'shrinking cannot happen in play'; the spawner orphan additionally requires live pre-warm records at the moment of this impossible shift. The finding describes a latent contract asymmetry with zero player-visible impact and no reachable leak — a documentation footnote, not a defect. The suggested hardening (direction flag / extended cleanup) is harmless but fixes nothing reachable.

### ✗ Spawner spawn queue stalls until the next chunk-boundary crossing when the 8192-entry ring fills（contracts）

The code mechanism is real: _enqueueChunkSpawns (src/world/spawner.js:728-736) abandons the top-up when _spawnQ.push fails, and retries only happen via _recomputeBand (boundary crossing / tier change / prewarm toggle). However, the failure scenario is not reachable. The finding's sizing premise ('~512 records x 72 placements') conflates the MAX_RECORDS pool capacity with the wanted set: wantK=72 applies only to the current tier band (~28-32 chunks at loadRadiusSim=96 / cellSizeSim=32 ≈ 2,300 placements); scenery is ~60 chunks x 8 ≈ 480 and pre-warm ~7 x 8 ≈ 56. The worst-case burst (tier change, all bands recomputed, scenery records upgraded 8→72) is ~2,300-3,000 entries — roughly a third of the 8,192-entry IntRing. Sustained accumulation requires the drain to stall, but store exhaustion is impossible (STORE_CAPACITY 8192 vs. occupancy ≤ ALIVE_TOTAL_BUDGET 4096, absorbs free their slots) and the aliveCount stall is a transient post-tier-change overshoot (~1,000 over budget) cleared in about a second by despawn (64/frame), leftover cleanup, and the sub-pixel sweep (up to 1,920 kills/s), while enqueue at speed cap adds only ~500 entries per ~1.5 s crossing vs. drain throughput of 3,840/s. Additionally, the 'permanently sparse while idle' symptom is overstated: _drainSpawn runs every update regardless of motion, so a full ring empties in ~2.1 s of standing still, and the residual un-enqueued tail heals on the first one-chunk movement. What remains is only an undocumented theoretical coupling between push-failure retry and recompute events, with no plausible in-game manifestation given a ring ~2.7x larger than the maximum possible burst.

**スキップ（理由付き）**:

- **InstancedPool single [min,max] dirty span degenerates to near-full uploads under churn (low)**: Multi-range updateRanges tracking (or 64-slot block bitmasks) is a structural change to the upload path with real regression risk for a steady-state-bandwidth-only win (~0.25-0.5 MB/frame worst case, no observed frame misses). The new shrink-on-drain fix already reduces span sizes for drained pools. Defer to a measured perf pass.
- **SpatialHash repack pays fixed O(16384) table cost per trigger (low)**: Amortized ~8k ops/frame bookkeeping, explicitly 'not a frame-breaker on desktop'. Making the counting sort content-proportional touches the core packed-region invariants; the safer LOOSE_CAP/TOMBSTONE retune changes broadphase timing behavior right after the cell-size fix. Skipped as low-value/medium-risk this pass.
- **Consumed-bitmask Map grows without bound within a tier (low)**: Slow leak (~hundreds of KB after minutes of pure roaming) already bounded by the tier-change cleanup in normal progression. The suggested LRU/distance eviction interacts with the determinism contract (evicted chunks regenerate consumed objects) — not worth the risk for a demo-length run.
- **Ball attachStuck / effects QuadPool bare instanceColor needsUpdate uploads (low)**: Full-buffer color uploads are 768B (ball families) and sub-KB (POP_CAP=96) — KB-scale per absorb, an internal-consistency nit rather than a budget problem. Adding a parallel per-family color dirty-range to ball.js is extra state in a hot file for negligible measured benefit.

## 最終バリデーション（ブラウザ実機）

Final validation of the patched Katamari build. npm run build passed cleanly (585.73 kB bundle). Drove the production preview at :4319 with agent-browser (session-isolated, synthetic KeyboardEvents against the window-level WASD listeners, absorbed count read from the #absorbed-value HUD node, sim time integrated in-page to correct for the 9-14fps software-GL environment). Title renders and START works. The first 30s playtest exposed a real gameplay regression the fixer's pickup-forgiveness patch had masked: only 2-4 absorbs per run. Root cause analysis of ballPhysics/tuning showed ACCEL_K=22 against FRICTION_PER_FRAME=0.92 puts terminal velocity at half the SPEED_K cap, making the cap dead code in all input states; fixed to 45 so the cap binds. A follow-up measured run was still under target, so pickup forgiveness was widened (K 0.45, max ratio 0.5, invariant vs ABSORB_RATIO preserved).
