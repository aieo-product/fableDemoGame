## 敵対的検証で棄却された指摘

### ✗ Finale descent handoff uses static TIERS[ti].moonAngSize instead of the live blended env.getMoonAngSize()（seams）

The code asymmetry is real: finale.js:370 computes the descent start distance from static TIERS[ti].moonAngSize while the same function uses the live blended direction via _getMoonDir/env.getMoonDirWorld (finale.js:371, 676-677), and environment.js:638-647 documents getMoonAngSize() as added precisely for this handoff — yet the finale never calls it (dead helper). However, the claimed visible size pop is unreachable in every scenario the finding cites, because at every reachable _beginDescent the blended _moonAngSize is exactly equal to TIERS[tierIndex].moonAngSize:

1. Normal play: the last TIER_UP fires at enterTrueRadius=150m (tiers.js:179, T5 'Skyline'), the palette fade lasts PALETTE_FADE_S=2.0s (tuning.js:252), and DESCENT triggers at MOON_GOAL_RADIUS_M=500m (tuning.js:35). A fade in flight at descent would require growing 150m->500m (37x volume) within 2 seconds — not achievable; MOON_CALL at 420m intervenes anyway. There is no tier boundary within ~350m of the goal.

2. 'Hysteresis-edge tier flips near the goal': refuted by topology. A down-flip needs tr < 150*0.9=135m, i.e. shrinking, which the game has no mechanic for (scaleManager.js:171 comment: 'shrinking cannot happen in play'; the down-loop is a float-edge guard). No flip can occur near 500m.

3. DEV watchdog force-descent (tr > 600m, finale.js:240-249): DEV-only, signals an already-broken trigger, and still has no fade in flight for the same topology reason — the last TIER_UP remains at 150m.

4. ?r= dev start (e.g. ?r=740, both thresholds in one frame): main.js:368-376 calls env.setTierPaletteImmediate(devTier) BEFORE the first frame. The first-frame TIER_UP cascade does call startPaletteFade, but _beginFade (environment.js:743-761) snapshots the current values — which are already the devTier endpoint — so the resulting fade is an identity fade for moonAngSize; _moonAngSize === TIERS[devTier].moonAngSize throughout.

5. The env-local NIGHT palette (index 6) reuses TIERS[5].moonAngSize (environment.js:89) and beginNightFade only runs in ASCENSION, after the handoff. resetWorld's setTierPaletteImmediate cancels any in-flight fade (_fadeDur=0, environment.js:673).

'Any future retune' is hypothetical, not a current defect. The suggested one-line fix is harmless, matches the documented intent of getMoonAngSize(), and would make the code robust against future retunes — worthwhile hygiene — but it changes nothing observable today. As a behavioral finding ('visible size pop ... reachable in practice') it is refuted; what remains is a latent-consistency/dead-helper note at low severity.

### ✗ Effects and Finale subscribe anonymous closures on the bus with no dispose/unsubscribe path at all（perf）

Code facts verified accurate: effects.js:307-332 subscribes 7 anonymous closures and finale.js:178-192 subscribes 2, with no stored references and no dispose(), while Backdrop/Environment/Bgm store handlers and bus.off() in dispose(). However, the finding describes no real behavior: (1) both classes are constructed exactly once at boot (main.js:228, 236) and restart goes through reset() only — confirmed no re-subscription path; (2) the leak scenarios are nonexistent infrastructure — there is no test directory and no vitest dependency in package.json, and no file uses import.meta.hot, so Vite performs full page reloads (fresh module-singleton bus) rather than HMR re-construction; (3) it is spec-compliant by design — the binding DESIGN-V2.md explicitly specifies dispose() for MoonView/Bgm/Backdrop but the frozen Finale interface (lines 98, 142-143) lists only reset(), and the effects.js entry (line 102) freezes its bus behavior as 'unchanged (already subscribed)'. The 'pinned Float32Array' consequence requires a second instance on the same bus, which no current or specified code path can create. What remains is a zero-cost future-proofing suggestion, not a perf defect — no allocation, leak, or frame cost occurs in any reachable execution of the shipped game.

### ✗ Rank table vs modeled pacing: S=300s sits below the ~345s clean-play baseline, and a normal first run lands C/D with zero in-game explanation of thresholds（gameplay）

Code reading is accurate (RANK_*_S only consumed in runStats.js:216; no UI threshold legend; rank stamp at screens.js:281), but the finding fails as a defect on three grounds. (1) Spec-compliant by design: docs/DESIGN-V2.md line 64 mandates the exact table 'S<=300/A<=400/B<=540/C<=720/else D', line 226 freezes the constants, and line 66 freezes the result-screen layout without a threshold legend — the code implements the binding spec verbatim. (2) Already tracked: DESIGN-V2.md line 273 makes re-tuning from >=3 real playthroughs a BINDING Phase-3 task ('one-line edits by design'), mirrored at tuning.js:102-103 — the finding's main remedy is already scheduled. (3) The central impact claim is contradicted by the project's own playtest data: spec line 64 records 'v1 averaged 6-8min to 500m; dash + flow should put good players ~5min'. 6-8 min = 360-480s lands A/B/C, and D requires >12 minutes — even the finding's own '8-12 min first-timer' gets C, not the claimed 'giant D'. The finding's ~345s arithmetic baseline is speculative; tuning.js:191-192 itself warns against tuning by arithmetic instead of measured playthroughs, and the measured v1 data makes S=300 match the spec's 'good player ~5min' target (the finding concedes S is defensible). What survives is a minor UX polish suggestion (next-rank-delta hint on the result screen, computable from RANK_*_S) — a reasonable enhancement to fold into the planned Phase-3 retune, not a medium gameplay defect.

### ✗ Dash gauge reads as a flat 4s cooldown — DASH_ABSORB_GAIN 0.03 is imperceptible against passive recharge（gameplay）

The finding's arithmetic is correct (passive 0.25/s from DASH_RECHARGE_S=4.0 at ballPhysics.js:119-121; 0.03/absorb at absorb.js:222-223; ~0.086/s at the documented T0 pace of ~0.35s/absorb, tuning.js:188-191), but the conclusion fails as a defect. (1) Spec-compliant by design: both values are explicitly frozen in the binding DESIGN-V2.md (line 58 and tuning block lines 213-214, Phase-0 value-frozen) — the code implements the spec exactly. (2) The asserted intent ("roll through dense areas to earn dashes" as the primary loop) appears nowhere in the spec; the absorb gain is documented as a minor one-line bonus on a timed recharge, so there is no broken feedback loop, only a deliberately conservative one. (3) The suggested fix conflicts with a documented binding constraint: DESIGN-V2.md line 274 states the 0.8s duration + 4s recharge bounds dash frequency to protect the 64/frame front-of-motion spawn budget against fog-wall pop-in at 2.2x cap; an absorb-dominant economy (gain 0.10-0.15) would enable ~2.3-3.5s refills precisely in dense areas, eroding that bound. (4) The "chime becomes noise" claim is overstated: DASH_READY is edge-emitted once per refill and a refill only follows a player-initiated dash, so its cadence is player-driven — standard cooldown-ready feedback. What remains is a subjective tuning observation (absorb gain is a ~25-34% modifier, gauge mostly reads as a cooldown) that matches the design's stated choice and is a one-line tuning edit subject to the spec's Phase-3 playtest retuning; at most a low-severity suggestion, not a medium gameplay defect.

### ✗ Toast queue-of-1 lets a rare pickup stomp the moon-call / 月へ向かえ telegraphs in their short ~7-10s window（gameplay）

Mechanics confirmed accurate: _showToast is queue-of-1 (hud.js:460-465), shared by rare (:295), moonCall (:333), and one-shot 月へ向かえ (:354-357); thresholds 420/500m and the ~12%-of-band-volume window check out. However the finding fails adversarial review on three grounds. (1) Spec-compliant by design: the binding DESIGN-V2.md explicitly mandates this exact contract — line 71: "#hud-toast (MOON_CALL / rare / 月へ向かえ, 3s, queue-of-1)", repeated at line 115 — so the shared queue-of-1 is the specified design, not an implementation bug. (2) Probability overstated: each moon toast is vulnerable only during its 3s display; with ~6-8 absorbs per 3s at designed pacing and RARE_CHANCE 0.002, the stomp probability is ~1.4% per window, ~3% per run — the finding conflates "rares are common per tier" with a rare absorb landing in a specific 3s window. (3) Impact cosmetic: moonCall is quadruple-redundant (toast + 0.5Hz sky-moon glow pulse persisting through CALLED + bgm shimmer swell + sfx pad), and the descent telegraph is the game's most dramatic visual (6s moon descent with angular-matched crossfade) plus the persistent 10Hz screen-edge arrow, which the toast stomp does not affect. The suggested priority mechanism is a legitimate Phase-3 polish idea, but as filed this is a design-preference note against explicitly specified behavior with rare occurrence and multi-channel backup.

### ✗ Share text: counter style inconsistency (こ vs コ) and ~100 weighted chars of unused headroom for a stronger hook（gameplay）

Spec-compliant by design. The share text in /Volumes/AIWorkSSD/AIWorkSpace/github/aieo-product/fableDemoGame/src/ui/screens.js:405-415 is a verbatim transcription of a frozen template specified character-for-character in the binding spec, docs/DESIGN-V2.md line 69 (§Xポスト), which itself contains `まきこんだ ${absorbed}こ・レア${rares}コ`. The こ/コ mixture is not a typo: the same pairing appears two more times in the spec (line 66 result-screen reveal "まきこんだ n こ + レア n コ" and line 62 "result shows レア n コ"), and follows the deliberate convention of matching the counter's script to the preceding word (hiragana まきこんだ→こ, katakana レア→コ), consistent with the game's kid-kana voice (さいごの大きさ, レアはっけん). Changing only the share text would desync it from the result screen UI that the spec mirrors. The second claim (unused ~109 weighted chars for an invite hook) is an enhancement proposal against a template the binding spec explicitly freezes with its length annotation "(~170/280 weighted chars — safe)"; honoring it would require a spec change, not a code fix. No malfunction, no incorrect data, no spec violation exists — the finding itself concedes the plumbing is correct.

**スキップ（理由付き）**:

- **Sky dome draw-order change (render after opaque world at far plane)**: Low severity and NOT low-risk: moving the dome to a positive renderOrder with depthTest + a forced far-plane gl_Position interacts with the backdrop silhouette ring (renderOrder -5, depthWrite:false, DoubleSide) and the blob shadow — done blind it would likely paint sky over the backdrop band.
