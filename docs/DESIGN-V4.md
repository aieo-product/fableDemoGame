# DESIGN-V4 — リアル東京（OSMストリートスケープ）+ モデル品質パス（確定差分仕様）

> 生成元: ultracode 設計ワークフロー `katamari-v4-design`。機械可読版: `docs/design/DESIGN-V4.json`

> データソースの法的判断: Google Maps Platform は規約上の派生データセット禁止により不採用。OpenStreetMap (ODbL) を採用し、帰属表示+ライセンスリンク+抽出日時+派生DB提供をリリースゲートとする。

## 概要

V4 "Real Tokyo" is a pure delta on live v3: every binding invariant survives (one-frame similarity rescale pixel-identity, seamlessness law, zero per-frame alloc, fixed 60Hz, deterministic spawn, static Cloudflare Pages, main.js integrator-only, DRAW_CALL_CAP 72, TRI_BUDGET 600k). Inside an explicitly-defined coverage geometry (detail disc r=500 game m around the Akihabara anchor + generated Shibuya and Asakusa patch rects, all owned by scripts/osm/geo.mjs), OpenStreetMap-derived voxel buildings replace the procedural band-3/4 fill as ABSORBABLE ObjectStore objects (FLAG_OSM=32, codes 94..109), placed at true geo positions under a uniform 1:5 horizontal / 1:2.5 vertical mapping. A build-time pipeline (resumable Overpass fetch with committed raw, deterministic converter with global dedupe + multipolygon ring assembly + merge-before-thin + road-corridor clearance baking, packed little-endian binary, expected ~0.8-1.0 MB raw / <=0.75 MB gz, hard cap 1536 KB gz enforced by a predeploy verify script with EXPECTED_COUNTS +/-20% assertions and a navigability gate) ships 2 fetched shards. A new OsmSpawner (curated origin/scale pattern verbatim, hard store-admission check at ALIVE_TOTAL_BUDGET-128) feeds the store; 2 new BatchedExtraPool batches (detail 2048 / large 1024, per-band membership boot-asserted against OSM_ALIVE_CAP) plus an OSM ground layer (roads/rail/parks batch + river mesh on the env water material; v3 authored bay quads KEPT) bring the honest draw ledger to 68 of cap 72. The 11 curated landmarks move to geo.mjs-GENERATED true positions (hand arithmetic banned; distance asserts in the validator); shop interior + exit street stay byte-identical so the validated opening minute cannot regress. The data race is closed by a one-way tier-2 deadline latch (fetch aborts, masks flip to procedural permanently, late data discarded; testable via ?osmdelay). Requirement 2 lands as a 12-id hero model pass (tri cap 350->600) + baked vertex AO + one shared-material rim term, each behind an independent kill switch and A/B-approved BEFORE the OSM world lands. ODbL compliance is a release gate: linked in-app credit, license link + extraction timestamp in README/manifest, committed raw + scripts as the derivative-database offer. Work splits into Phase 0 + 4 zero-overlap streams (P pipeline / W world / R render / C content), integration order P -> C -> R -> W.

## データパイプライン

BUILD-TIME ONLY (deploys never touch Overpass; raw fetches AND converted bins are committed). Scripts under scripts/osm/, run via `npm run osm:fetch` / `osm:build` / `osm:verify`.

0) BASELINES (corrected, the binding ground truth committed as EXPECTED_COUNTS in geo.mjs, re-derived by `out count;` queries per fetch set at fetch time): detail square (35.6759,139.7417,35.7211,139.7969) = 58,155 buildings (58,035 ways + 120 multipolygon relations), density 2,321/km^2-real; roads+rail full bbox = 16,712 ways; water+parks = 4,272 elements; towers (height>=50 OR levels>=13, full bbox) = 1,946. verify-tokyo-data.mjs asserts manifest per-band counts within +/-20% of EXPECTED_COUNTS so a silent Overpass data shift can never invalidate the alive caps or the pacing model. All prose numbers below derive from these baselines; the manifest is authoritative at integration time.

1) FETCH (scripts/osm/fetch-osm.mjs). RESUMABLE: a cell whose data/osm-raw/<kind>-<cell>.json.gz already exists is skipped (incremental commits, crash-safe). Rate-limit protocol: poll /api/status before each request and sleep until a slot is free; HTTP 429 or a Dispatcher_Client rate_limited body sleeps the advertised retry-after WITHOUT consuming one of the 3 hard retries; bulk runs default to the kumi mirror https://overpass.kumi.systems/api/interpreter with overpass-api.de as fallback; >=2 s spacing floor. Raw responses committed tag-pruned (jq keeps only: building, height, building:levels, building:* type keys, highway, railway, tunnel, natural, water, waterway, leisure, name). Fetch sets ({{bbox}} = S,W,N,E per cell from geo.mjs, cells DERIVED as the 1x1 km cell set covering the coverage geometry — never hand-listed):

2) CONVERT (scripts/osm/build-tokyo-bin.mjs; deterministic: same raw -> byte-identical bins, asserted by osm:verify double-run). PIPELINE ORDER IS BINDING:
   step 0 DEDUPE+ASSEMBLE: load ALL raw files into one map keyed by (type,id) — dedupes ways straddling cell boundaries AND across fetch sets; for buildings present in both detail and tower sets, the TOWER record wins and the detail record is dropped. Multipolygon relations: stitch member outer ways end-to-end into closed rings (osmtogeojson-style endpoint matching); hull the outer ring(s) only; ignore inner rings (voxel boxes need no holes); member ways consumed by a relation are removed from the standalone-way set (no double count). Unstitchable relations: drop + log, never crash; verify reports the count.
   step 1 PROJECT via geo.mjs toGame(lat,lon): x=(lon-ANCHOR_LON)*M_PER_DEG_LON*OSM_HORIZ_K, z=(ANCHOR_LAT-lat)*M_PER_DEG_LAT*OSM_HORIZ_K (+X east, +Z south).
   step 2 COVERAGE CLIP: keep a building iff its footprint centroid is inside the coverage geometry (disc r=500 game m + the two patch rects, all defined ONLY in geo.mjs); tower-set records are exempt (full bbox). verify asserts zero detail records outside coverage.
   step 3 EXCLUSIONS (baked, verify-asserted empty): shop interior rect +2 m margin; curated 中央通り strip x[0,25]xz[-190,190]; circles of dioramaR*1.2 at each RECONCILED landmark position (computed by geo.mjs, dedupes the real ラジオ会館/109/東京駅/議事堂/ドーム/雷門 footprints in BOTH shards); Skytree base circle r=110 game at the generated SKYTREE_POS.
   step 4 OBB: convex hull -> min-area OBB (rotating calipers). NO two-box decomposition in format v1 (COMPOSITE/FOLLOW cut — see gameplay); convert MEASURES the would-have-decomposed rate (polyArea/obbArea<0.65 AND obbArea>80 game-m^2) and logs it; revisit only if >3% of total area.
   step 5 HEIGHT: h_real = parseNum(height) || levels*3.2 || default by type (house 6.5 / retail 9 / apartments 12 / office 18 / temple-shrine 8 m); h_game = h_real*OSM_HEIGHT_K. parseNum strips unit suffixes ("50 m"). Tower threshold applied here (>=50 m or >=13 levels keeps tower-set records; below-threshold tower-set records inside coverage fold into the detail set, outside coverage are dropped).
   step 6 QUANTIZE (the voxel law): center 0.05 m, w/d 0.25 m, h 0.5 m, yaw pi/128 — quantization IS both the aesthetic and the compression.
   step 7 MERGE (BEFORE thinning and banding): adjacent buildings sharing >=60% of an edge, same type class, both w,d < 4 game m -> one OBB, re-quantized (FLAG merged; expected -15-25% in dense 下町 grids).
   step 8 CLEARANCE BAKE (navigability law): build road corridors from ALL fetched road classes inside coverage (incl. residential/unclassified): corridor halfwidth = roadWidth_game/2 + OSM_CLEARANCE_M (1.0 game m = 5 m real). Any building OBB intruding into a corridor is INSET along the intruding axis toward its own centroid (max 30% of that dimension); if still intersecting after max inset, or post-inset min(w,d) < 0.5 game m, the building is DROPPED. Deterministic, geometry-only.
   step 9 BAND + THIN: r_eff = 0.5*sqrt(w^2+d^2+h^2) game m; DROP r_eff < 1.2. Band cut (re-derived so typical houses are band 3: an 8x8x6.5 m house -> r_eff 1.72): [1.2,1.6)->2, [1.6,10)->3, [10,60)->4, >=60->5. collisionScale = clamp(0.5*sqrt(w^2+d^2)/r_eff, 0.35, 1.0). Deterministic thinning: keep iff mulberry32(wayId)() < OSM_KEEP_K_BY_BAND[band]. verify prints the band histogram; OSM_ALIVE_CAP is re-cut from it in Phase 3.
   step 10 TYPE->code (16 archetypes, codes 94..109, frozen Phase 0): osm_house, osm_shop_low, osm_zakkyo, osm_office_mid, osm_office_tower, osm_apartment_tower, osm_hotel, osm_school, osm_temple, osm_shrine, osm_station, osm_warehouse, osm_parking, osm_merged_block, osm_tower_generic, osm_stepped_roof. tint = type palette row hashed by wayId.
   step 11 ROADS (shipped set): full bbox motorway|trunk|primary|rail; secondary|tertiary ONLY inside detail coverage (the outer-tile size lever). Width by class (real 25/18/14/10/8 m /5), Douglas-Peucker eps=1 game m (eps=2 outside coverage), clipped to ground tiles, split major/minor for LOD. WATER/PARKS: rings DP-simplified eps=1 game m FIRST, then earcut-triangulated at build time (u16 indices; no runtime earcut); parks kept iff real area >= 5,000 m^2. Baked layer y-offsets (see rendering): parks +0.02, minor roads +0.04, major roads/rail +0.06 game m.
   step 12 EMIT tiles + manifest (incl. per-band counts, band histogram, extraction timestamp, attribution + ODbL license URL).

3) BINARY TILE FORMAT v1 (little-endian, public/assets/tokyo/): tokyo-v4-core.bin = detail tiles (grid 100 game m); tokyo-v4-outer.bin = tower tiles (400 game m grid) + ground tiles (200 game m grid); tokyo-v4-manifest.json = {version, shard gz sizes, per-band counts, band histogram, tile index summary, extractionDate, attribution:"© OpenStreetMap contributors", license:"ODbL", licenseUrl:"https://opendatacommons.org/licenses/odbl/"}.
   Header (16 B): magic 'FKT4' u32, version u16, sectionCount u16, flags u32, reserved u32. Section entry (16 B): type u8 (1=detail,2=tower,3=road,4=poly), pad u8, tileX i16, tileZ i16, count u16, byteOffset u32, byteLen u32.
   DETAIL (10 B): cx u16, cz u16 (tile-local, 0.05 m), w u8, d u8 (0.25 m, max 63.75), h u8 (0.5 m, max 127.5), yaw u8 (pi/128), type u8 (bits 0-4 = code-94; bit 5 = MERGED; bits 6-7 reserved — COMPOSITE/FOLLOW removed), tint u8.
   TOWER (12 B): cx u16, cz u16, w u8, d u8, h u16 (0.25 m), yaw u8, type u8, tint u8, pad u8.
   ROAD (8+4(n-1) B): classAndCount u8 (class:3, n:5, n<=31, longer ways pre-split), width u8 (0.25 m), pad u16, firstX u16, firstZ u16 (0.1 m tile-local), then (n-1)x(dx i16, dz i16 at 0.1 m).
   POLY (6+4n+2t B): kind u8 (water/park, layer bits), pad u8, n u16, t u16 (tri index count) — u16 BOTH (隅田川/上野公園 exceed 255 vertices even simplified; verify prints the max-n histogram to prove the u16 ceiling safe), verts as ROAD, tri indices u16.
   SIZE MATH (corrected baselines, labeled ESTIMATE until manifest measures): detail post-clip/merge/thin ~28-30k x10 B ~300 KB; towers ~2k x12 B ~24 KB; shipped roads (major full-bbox + secondary/tertiary in-coverage) ~6-7k ways x ~36 B ~250 KB; polys ~1.2k post-filter x ~150 B avg ~180 KB; tables/headers ~15 KB -> ~0.8-1.0 MB raw -> ~0.55-0.75 MB gz. HARD CAP OSM_DATA_BUDGET_GZ_KB=1536 enforced by scripts/osm/verify-tokyo-data.mjs (non-zero exit over budget). verify also asserts/reports: EXPECTED_COUNTS +/-20%; zero duplicate source ids across both shards; zero records outside coverage; zero exclusion-zone violations; landmark dedupe report; unstitchable-relation count; height-defaulted fraction histogram by district; POLY max-n histogram; band histogram; double-run byte-identity; NAVIGABILITY REPORT (rasterize footprints at 0.5 game m, erode free space by ball radius, flood-fill from seed (30,0) outside the curated strip, print reachable road-corridor fraction per ball-radius bracket {1,3,8,20}; Phase-3 GATE: >=95% for brackets 1 and 3). Wired into predeploy next to verify-donack-assets.sh.

4) RUNTIME LOAD (src/world/osmWorld.js): fetch() both shards in parallel at title screen (immutable cache via public/_headers); decode into flat typed arrays in one pass (~5-10 ms, title-screen work — documented exemption alongside the thumbnail pre-render); emit EVT.OSM_READY. ## レンダリング統合

DECISION — NO separate building scenery tile mesh: fog far (55r) < load radius (asserted since v1), so OSM buildings exist ONLY as ObjectStore objects (bands 2..5 via the live window), exactly like procedural fill. Collision/absorb/burial/subpixel semantics untouched; no double representation. Mid-rises up to r_eff 10 are band 3, so the Akihabara street wall is present as N+1 scenery already at T2.

OSM BUILDING POOLS (src/render/osmPools.js, Stream R): two BatchedExtraPool instances (the proven v3 class, 1 draw each via WEBGL_multi_draw): osm-detail (cap OSM_POOL_DETAIL_CAP=2048; members = the small/mid archetypes used by bands 2-3) and osm-large (cap OSM_POOL_LARGE_CAP=1024; members = tower/block archetypes used by bands 4-5). POOL MEMBERSHIP IS PER-BAND AND FROZEN PHASE 0, with the boot-asserted invariant: sum of OSM_ALIVE_CAP over each pool's member bands <= pool capacity (b2+b3 = 192+1536 = 1728 <= 2048; b4+b5 = 768+128 = 896 <= 1024) — slot exhaustion is structurally impossible (this fixes the draft's 512-cap underrun vs 928 worst-case demand). Per-instance NON-UNIFORM scale: extraPools.setTransform and instances.setTransform gain optional (sx,sy,sz) (Matrix4.compose supports it; fade animators store/multiply the scale triple; existing callers default uniform — zero behavior change). OSM archetype geometries use a UNIT-BOX convention (half-extents 1; catalog flag unitBox:true) so render scale=(w/2,h/2,d/2) while store radius stays r_eff; the unit-sphere convention stays binding for all other archetypes. NORMALS CONSTRAINT (boot-asserted at geometry build): every unitBox:true geometry's normals must be axis-aligned (+/-X/Y/Z) because BatchedMesh applies no inverse-transpose — non-uniform scale would mislight sloped faces. Consequence: ALL 16 OSM archetypes are flat/stepped boxes in v1; temple/shrine ship flat-roofed with vermilion/wood banding (sloped-roof fixed-aspect variants are a documented post-ship option). rescaleAll(S)/rebaseAll(sx,sz)/reset() come free from the class; visible=false when empty.

OSM GROUND (src/render/osmGround.js, Stream R): one parent THREE.Group whose transform is a PURE FUNCTION of (worldScale, originShift): group.scale = 1/worldScale (OSM vertex units are game meters), group.position = -origin — the one-frame rescale is a uniform similarity on one node (pixel-identical by construction, same trick as backdrop); rebase is one position write. Children: (1) ground BatchedMesh — road/rail ribbons (dark grays by class) + park patches (greens), vertex-colored, one MeshBasicMaterial({vertexColors:true}), fog ON, renderOrder after the ground shader (= 1 draw). Intra-batch coplanarity is solved by the CONVERT-BAKED layer y-offsets (parks +0.02 / minor +0.04 / major+rail +0.06 game m) — baked offsets scale with the similarity so the rescale invariant holds; polygonOffset(-1,-1) remains only as belt-and-braces vs the terrain. (2) river/water mesh sharing environment.js's bay-water material via new env.getWaterMaterial() (= 1 draw). The v3 authored bay quads are KEPT — OSM water is rivers/ponds/moats only (the river mesh is ADDITIVE; no replacement claimed, no freed slot). Tile lifecycle: tiles within max(fogFarSim*1.1, load floor) get addGeometry'd, <=OSM_GROUND_TILE_BUILDS_PER_FRAME(2) builds/frame; ring-exit deletes; minor-road sections only within 0.5*fogFar (ground LOD); at T5+ all ground tiles resident at major-roads-only LOD (~40k tris of flat geometry).

DRAW-CALL LEDGER v4 (honest, no double counting): v3 worst 64 (40 world + 8 stuck + 6 fixed + 1 backdrop + 2 skytree + 1 terrain + 2 water+quay [bay KEPT] + 4 extra pools) + 2 OSM building batches + 1 OSM ground batch + 1 OSM river mesh = 68. DRAW_CALL_CAP STAYS 72. Pre-approved over-cap levers in order: (1) merge river into the ground batch as a water vertex-color class sharing a water-ish flat shade (-1); (2) hide osm-large outside band-4/5 windows (-1).

TRI LEDGER: OSM alive worst ~3,072 pool slots x ~36 avg tris (box 12, stepped <=72) ~110k + ground ~40k; band-3/4 procedural pool tris MIGRATE (near-empty inside coverage), not add. With the hero pass (+~110k modeled worst), worst scene stays under TRI_BUDGET 600k — measured by renderer.info parked at 秋葉原 r=4 and 丸の内 r=40 in the Phase-3 profile pass; first lever HERO_TRI_CAP 600->500.

## リスク

- Real-density pacing at the CORRECTED 2.8x multiplier: if KEEP_K + the growthKForObjR taper under-compensate, T3/T4 collapse into minutes-long cascades. Gates: pacing-model reads the manifest (never prose) BEFORE integration; >=3 driven runs binding; levers are data-only pipeline re-runs. The +/-20% EXPECTED_COUNTS assert prevents a silent upstream data shift from invalidating the retune.
- Navigability is the single biggest does-it-play risk: clearance baking is geometry surgery on real footprints, and over-aggressive insets could make 下町 look gap-toothed while under-aggressive ones wedge the ball. The verify navigability gate (>=95% at brackets 1/3) is the hard line; the visual cost of insets is reviewed on the Stream P density report screenshots before Phase 3.
- Population ceiling at the coverage boundary now relies on RUNTIME enforcement (admission check) rather than static arithmetic — correct by construction but it means OSM streetscape visibly thins under worst-case pressure. The boundary park test (300 frames at r~4/40) plus the 0.9-budget DEV warn are the gates; pre-approved lever: b3 cap 1536->1200 + KEEP_K[3] 0.5.
- OSM data quality: ~87% of buildings lack height/levels tags — type-default heuristics drive most of the skyline. Mitigation: verify height-defaulted histogram reviewed against known districts; committed raw freezes the dataset so tuning is reproducible.
- Failure-path determinism caveat: a fetch failure (or the tier-2 deadline) flips bands 3/4 back to procedural — a failed-run differs from a normal run at the same seed. Accepted, one-way, documented; testable via ?osmdelay; ~80 s of slack before band 3 matters.
- BatchedMesh on low-end Android at 2048+1024 instances: WEBGL_multi_draw absence degrades perf not correctness. Gate: real-Android profile; fallback levers: split osm-detail (+1 draw, still <=72) or cap detail alive.
- Non-uniform instance scale is new to the instancing path: any code deriving radius from matrix scale breaks. Stream R audit item + DEV assert (store radius vs pool scale on alloc); the axis-aligned-normals boot assert prevents the lighting half of the same bug class.
- Rim+AO+palette change EVERY approved object: mitigated by 3 separable commits, independent kill switches, and the A/B sheet approved BEFORE the OSM world lands (bisectable sign-off). Rim is similarity-invariant by construction but the KeyR diff test remains the proof.
- Landmark moves change travel legs (Diet->Tower ~390 m; Tower->Skytree ~1,640 m; Rainbow Bridge relocates SSW at its TRUE ~160 game m span): validator distance asserts + ladder/travel reprint + Phase-3 runs decide any dioramaR nudges; all single-constant edits. Hand-typed geo values are banned — geo.mjs generates, the validator cross-checks.
- Committed raw (~8-12 MB gz incl. clearance-only residential roads) bloats the repo; shipped data ~0.55-0.75 MB gz vs hard cap 1536 KB enforced by verify. Title-screen #osm-progress keeps slow-network UX honest.
- ODbL is a RELEASE GATE, not a docs nicety: linked in-app credit (osm.org/copyright), license link + extraction timestamp in README/manifest, and the derivative-database offer (public data/osm-raw/ + scripts/osm/) must all exist before deploy.
- Absorbed OSM buildings skip the world-slot fly-on animation (EXTRA-pool precedent); at T3-T5 buildings dominate absorbs. If the pop feels flat, the documented post-ship polish lever is a 0.15 s BatchedMesh slot animation pre-attach.
- Opening-minute regression remains the highest-trust invariant: shop/strip/gutter data byte-identical, OSM excluded from x[0,25]xz[-190,190], and the integration checklist still mandates the full ?at=shop T0/T1 validator + visual run before deploy.
